## static — 三种用法，三个含义

### 1. 修饰局部变量（函数内）
```c
void counter(void) {
    static int count = 0;  // 只初始化一次，生命周期=整个程序运行期
    count++;
    printf("%d\n", count);
}
```
- 存储在 DATA/BSS 段（不是栈），函数返回后值不丢失
- 只在第一次调用时初始化，后续调用跳过初始化
- **面试常问**：不加 static 会怎样？→ 每次进函数都清零，存在栈上

### 2. 修饰全局变量/函数（文件内）
```c
// file1.c
static int g_flag = 0;     // 仅 file1.c 内可见
static void helper(void) { } // 仅 file1.c 内可调用
```
- 限制作用域为本文件，外部 `extern` 不到
- **面试常问**：为什么要用？→ 模块封装，避免命名冲突（嵌入式项目多文件编译必备）

### 3. 修饰函数内静态变量 vs 全局变量
| 对比项 | static 局部 | static 全局 |
|--------|-------------|-------------|
| 作用域 | 函数内 | 本文件内 |
| 生命周期 | 程序运行期 | 程序运行期 |
| 存储位置 | DATA/BSS | DATA/BSS |
| 初始化 | 首次调用时 | main 之前 |

### 面试追问：static 变量存在哪？
在 DATA 段（已初始化）或 BSS 段（未初始化/初始化为0），不在栈上。

## const — 修饰谁，谁就不能改

### 规则：const 修饰它右边最近的类型名
```c
const int a = 10;    // a 不可改
int const a = 10;    // 同上

const int *p;        // *p 不可改（指向的内容只读）
int const *p;        // 同上
int * const p;       // p 不可改（指针本身只读）
const int * const p; // 指针和内容都只读
```

### 记忆口诀（从右往左读）
- `int * const p` → p is a const pointer to int
- `const int *p` → p is a pointer to const int

### 嵌入式场景
```c
// ROM 中的查找表，用 const 让编译器放到 FLASH 而不是 RAM
const uint8_t sine_table[256] = { ... };

// 函数参数声明只读，防止误改
void send_data(const uint8_t *buf, uint16_t len);
```

### 面试必问：const 和 #define 的区别
| | const | #define |
|---|-------|---------|
| 类型检查 | 有 | 无（纯文本替换） |
| 调试可见 | 有符号表 | 无（预处理后消失） |
| 存储 | 占用内存 | 不占内存（编译期替换） |
| 作用域 | 有 | 无（文件级） |

## volatile — 告诉编译器"别优化我"

### 三条铁律——以下情况必须用 volatile：

**1. 中断服务函数中修改的全局变量**
```c
volatile uint8_t flag = 0;  // 主循环读，ISR 写

void EXTI_IRQHandler(void) {
    flag = 1;  // ISR 中修改
}

void main(void) {
    while (1) {
        if (flag) {  // 不加 volatile，编译器可能把 flag 优化到寄存器，永远读不到 ISR 的修改
            flag = 0;
            do_something();
        }
    }
}
```

**2. 硬件寄存器**
```c
#define GPIOA_ODR  (*(volatile uint32_t *)0x4001080C)
// 不加 volatile，编译器可能优化掉连续两次写同一寄存器的操作
```

**3. 多线程共享变量（FreeRTOS 任务间）**
```c
volatile int shared_flag;  // 任务 A 写，任务 B 读
// 但注意：volatile 不保证原子性！多字节变量仍需配合临界区/互斥锁
```

### 面试必问：volatile 能保证原子性吗？
**不能！** volatile 只禁止编译器优化（强制每次从内存读），不提供原子操作。对于 32 位系统上的 `volatile int` 读是原子的，但 `volatile uint64_t` 在 32 位系统上的读写就不是原子的。真正需要原子性时用关中断、互斥锁或原子操作指令。

### 面试必问：const 和 volatile 能同时用吗？
能。经典场景：**只读状态寄存器**。
```c
const volatile uint32_t *status_reg = (uint32_t *)0x40000000;
// const：程序不能写这个寄存器
// volatile：硬件可能随时改变它，每次都要读
```

## extern — 声明"这个东西在别的文件里"

```c
// file1.c — 定义
int g_counter = 0;
void init_system(void) { ... }

// file2.c — 声明（不加 extern 就会重复定义，链接报错）
extern int g_counter;
extern void init_system(void);
```

### 面试常问：声明 vs 定义
- **声明**：告诉编译器"有这个东西，不分配空间"（extern 变量声明、函数原型）
- **定义**：分配实际内存空间（变量定义、函数体）
- 变量可以多次声明，只能定义一次（C 语言「单一定义规则」）

## typedef vs #define — 面试高频比较

```c
typedef char *pChar;      // 类型别名，编译器处理
#define pChar2 char *     // 文本替换，预处理器处理

pChar  a, b;   // char *a, *b;   → 两个都是指针
pChar2 c, d;   // char *c, d;    → c 是指针，d 是 char！（陷阱）
```

### 区别
| | typedef | #define |
|---|---------|---------|
| 处理阶段 | 编译期 | 预处理期 |
| 作用 | 给类型取别名 | 文本替换 |
| 作用域 | 有 | 无 |
| 多变量声明 | 行为正确 | 可能出 bug |
